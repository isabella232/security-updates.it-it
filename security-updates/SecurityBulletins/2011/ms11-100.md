Security Bulletin

Bollettino Microsoft sulla sicurezza MS11-100 - Critico
=======================================================

Alcune vulnerabilità in .NET Framework possono consentire l'acquisizione di privilegi più elevati (2638420)
-----------------------------------------------------------------------------------------------------------

Data di pubblicazione: giovedì 29 dicembre 2011 | Aggiornamento: martedì 10 luglio 2012

**Versione:** 1.6

### Informazioni generali

#### Riepilogo

Questo aggiornamento per la protezione risolve una vulnerabilità divulgata pubblicamente e tre vulnerabilità segnalate privatamente in Microsoft .NET Framework. La vulnerabilità con gli effetti più gravi sulla protezione può consentire l'acquisizione di privilegi più elevati se un utente malintenzionato non autenticato invia una richiesta Web appositamente predisposta al sito di destinazione. Un utente malintenzionato in grado di sfruttare questa vulnerabilità può intraprendere un'azione qualsiasi nel contesto di un account esistente sul sito ASP.NET, compresa l'esecuzione di comandi non autorizzati. Per sfruttare questa vulnerabilità, un utente malintenzionato deve essere in grado di registrare un account sul sito ASP.NET e deve conoscere un nome utente esistente.

Questo aggiornamento per la protezione è considerato di livello critico per Microsoft .NET Framework 1.1 Service Pack 1, Microsoft .NET Framework 2.0 Service Pack 2, Microsoft .NET Framework 3.5 Service Pack 1, Microsoft .NET Framework 3.5.1 e Microsoft .NET Framework 4 su tutte le edizioni supportate di Microsoft Windows. Per ulteriori informazioni, vedere la sottosezione **Software interessato e Software non interessato** in questa sezione.

L'aggiornamento per la protezione risolve le vulnerabilità correggendo il modo in cui .NET Framework gestisce le richieste appositamente predisposte e il modo in cui ASP.NET Framework autentica gli utenti e gestisce il contenuto memorizzato nella cache. Per ulteriori informazioni sulle vulnerabilità, cercare la voce specifica nella sottosezione Domande frequenti della sezione **Informazioni sulle vulnerabilità**.

Questo aggiornamento per la protezione risolve anche la vulnerabilità descritta per la prima volta nell'[Advisory Microsoft sulla sicurezza 2659883](http://technet.microsoft.com/security/advisory/2659883).

**Raccomandazione.** Se la funzionalità Aggiornamenti automatici è abilitata, gli utenti non devono intraprendere alcuna azione, poiché questo aggiornamento per la protezione viene scaricato e installato automaticamente. Gli utenti che non hanno attivato la funzionalità Aggiornamenti automatici devono verificare la disponibilità di aggiornamenti e installare questo aggiornamento manualmente. Per informazioni sulle opzioni di configurazione specifiche relative agli aggiornamenti automatici, vedere l'[articolo della Microsoft Knowledge Base 294871](http://support.microsoft.com/kb/294871).

Agli amministratori e alle organizzazioni o agli utenti finali che desiderano installare manualmente questo aggiornamento per la protezione, Microsoft consiglia di applicare l'aggiornamento utilizzando il software di gestione degli aggiornamenti o verificando la disponibilità di questi tramite il servizio [Microsoft Update](http://www.update.microsoft.com/microsoftupdate/v6/vistadefault.aspx?ln=it-it).

Vedere anche la sezione, **Strumenti e informazioni sul rilevamento e sulla distribuzione**, riportata di seguito nel presente bollettino.

**Problemi noti.** L'[articolo della Microsoft Knowledge Base 2638420](http://support.microsoft.com/kb/2638420) descrive i problemi attualmente conosciuti che gli utenti potrebbero riscontrare durante l'installazione di questo aggiornamento per la protezione. L'articolo illustra inoltre le soluzioni consigliate in grado di risolvere questi problemi.

Le altre versioni o edizioni non sono interessate dalla vulnerabilità o sono al termine del ciclo di vita del supporto. Per informazioni sulla disponibilità del supporto per la versione o l'edizione del software in uso, visitare il [sito Web Ciclo di vita del supporto Microsoft](http://support.microsoft.com/common/international.aspx?rdpath=gp;%5Bln%5D;lifecycle).

Domande frequenti sull'aggiornamento per la protezione
------------------------------------------------------

<span></span>
**Perché Microsoft ha rivisto il bollettino in data** **22maggio** **2012?**
Microsoft ha rivisto questo bollettino per comunicare una modifica relativa al rilevamento per Microsoft .NET Framework 2.0 Service Pack 2 al fine di correggere un problema di installazione. Si tratta di una modifica solo al rilevamento. Non sono previsti aggiornamenti ai file sull'aggiornamento per la protezione. I clienti che hanno già aggiornato i propri sistemi non devono eseguire ulteriori operazioni.

**Questo bollettino si riferisce a MS12-035, alcune vulnerabilità in .NET Framework possono consentire l'esecuzione di codice in modalità remota (2693777)?**
Sì. L'aggiornamento KB2656353, offerto in questo bollettino, risolve anche le vulnerabilità CVE-2012-0160 e CVE-2012-0161 descritte in [MS12- il 035](http://go.microsoft.com/fwlink/?linkid=246970).

**Microsoft .NET Framework 3.5 e Microsoft .NET Framework 4.5** **in Windows** **Developer Previewsono interessati da questa vulnerabilità?**
Sì. Questa vulnerabilità è stata segnalata dopo il rilascio di Windows Developer Preview. Ai clienti che utilizzano il software Developer Preview si consiglia di scaricare e applicare l'aggiornamento ai sistemi in uso. Per ulteriori informazioni, compresi i collegamenti dell'Area download Microsoft, vedere gli articoli della Microsoft Knowledge Base seguenti:

| Software interessato                                      | Articolo della Microsoft Knowledge Base                                                    |
|-----------------------------------------------------------|--------------------------------------------------------------------------------------------|
| Microsoft .NET Framework 3.5 in Windows Developer Preview | [Articolo della Microsoft Knowledge Base 2656360](http://support.microsoft.com/kb/2656360) |
| Microsoft .NET Framework 4.5 in Windows Developer Preview | [Articolo della Microsoft Knowledge Base 2656354](http://support.microsoft.com/kb/2656354) |

Gli aggiornamenti per la protezione sono disponibili da [Microsoft Update](http://www.update.microsoft.com/microsoftupdate/v6/vistadefault.aspx?ln=it-it) e [](http://www.update.microsoft.com/microsoftupdate/v6/vistadefault.aspx?ln=it-it)[Windows Update](http://www.update.microsoft.com/microsoftupdate/v6/vistadefault.aspx?ln=it-it)</a>.

**L'aggiornamento include modifiche di funzionalità relative alla sicurezza?**
Sì. Oltre alle modifiche elencate nella sezione Informazioni sulle vulnerabilità di questo bollettino, questo aggiornamento contiene modifiche funzionali ad ASP.NET Framework. L'aggiornamento limita il numero massimo di chiavi, file e membri JSON dei moduli a 1000 in una richiesta. Questa modifica fa sì che le applicazioni ASP.NET rifiutino le richieste in cui il numero dei suddetti elementi è superiore a 1000. I client che effettuano richieste di questo tipo verranno respinti e verrà visualizzato un messaggio di errore nei relativi browser (solitamente l'errore 500). Questo limite può essere configurato per ciascuna applicazione. Vedere [l'articolo della Microsoft Knowledge Base 2661403](http://support.microsoft.com/kb/2661403).

L'aggiornamento modifica inoltre il formato dei ticket di autenticazione dei moduli in modo tale che potrebbe risultare incompatibile con la versione precedente dei ticket di autenticazione dei moduli. Ad esempio, dopo aver installato questo aggiornamento, negli ambienti Web farm in cui alcuni server vengono aggiornati e altri no, in alcuni casi i server aggiornati potrebbero generare ticket di autenticazione dei moduli incompatibili sui server non aggiornati. Vedere [l'articolo della Microsoft Knowledge Base 2661404](http://support.microsoft.com/kb/2661404).

**Dove si trova la sezione Informazioni sui file?**
Fare riferimento alle tabelle contenute nella sezione **Distribuzione dell'aggiornamento per la protezione** per individuare la sezione Informazioni sui file.

**Come è possibile determinare quale versione di Microsoft .NET Framework è installata nel proprio sistema?**
È possibile installare ed eseguire più versioni di .NET Framework su un sistema ed installarle in un ordine qualsiasi. Esistono diversi modi per determinare quali versioni di .NET Framework sono attualmente installate sul proprio sistema. Per ulteriori informazioni, vedere l'[articolo della Microsoft Knowledge Base 318785](http://support.microsoft.com/kb/318785).

**Qual è la differenza tra .NET Framework 4 e .NET Framework 4 Client Profile?**
Le versioni 4 dei redistributable package .NET Framework sono disponibili in due profili: .NET Framework 4 e .NET Framework 4 Client Profile. .NET Framework 4 Client Profile è un sottoinsieme di .NET Framework 4, che è ottimizzato per le applicazioni client. Fornisce la funzionalità per la maggior parte delle applicazioni client, comprese le funzioni di Windows Presentation Foundation (WPF), Windows Forms, Windows Communication Foundation (WCF) e Clickonce. Ciò consente una distribuzione più rapida e un pacchetto di installazione di minori dimensioni per applicazioni che mirano a .NET Framework 4 Client Profile. Per ulteriori informazioni, consultare l'articolo di MSDN, [.NET Framework Client Profile](http://msdn.microsoft.com/en-us/library/cc656912.aspx).

**Sono elencati due aggiornamenti per la versione di Microsoft .NET Framework installata nel proprio sistema. È necessario installare entrambi gli aggiornamenti?**
Sì. Applicare tutti gli aggiornamenti offerti per il software installato nei sistemi in uso.

**È necessario installare in una sequenza particolare questi aggiornamenti per la protezione?**
No. È possibile installare più aggiornamenti per una versione di .NET Framework in un ordine qualsiasi. È consigliabile installare più aggiornamenti per versioni diverse di .NET Framework secondo un ordine che va dal numero di versione più basso a quello più alto, anche se non è necessario seguire quest'ordine.

**È stato installato .NET Framework 3.0 Service Pack 2; questa versione non è elencata nel software interessato in questo bollettino. È necessario installare un aggiornamento?**
Questo bollettino descrive una vulnerabilità nel livello di funzionalità di .NET Framework 2.0 e .NET Framework 4. Il programma di installazione di .NET Framework 3.0 Service Pack 2 è collegato alla configurazione di .NET Framework 2.0 Service Pack 2, quindi l'installazione del primo programma comporta anche l'installazione del secondo. Quindi, i clienti che hanno installato .NET Framework 3.0 Service Pack 2 devono installare gli aggiornamenti per la protezione per .NET Framework 2.0 Service Pack 2.

**È stato installato .NET Framework 3.5 Service Pack 1. È necessario installare eventuali aggiornamenti aggiuntivi?**
Questo bollettino descrive una vulnerabilità nel livello di funzionalità di .NET Framework 2.0 e di .NET Framework 4. Il programma di installazione di .NET Framework 3.5 Service Pack 1 è collegato sia alla configurazione di .NET Framework 2.0 Service Pack 2 che di .NET Framework 3.0 Service Pack 2. Quindi, i clienti che hanno installato .NET Framework 3.5 Service Pack 1 devono installare anche gli aggiornamenti per la protezione per .NET Framework 2.0 Service Pack 2.

**Se si utilizza una versione precedente del software discusso nel presente bollettino, come bisogna comportarsi?**
I prodotti software elencati in questo bollettino sono stati sottoposti a test per determinare quali versioni siano interessate dalla vulnerabilità. Le altre versioni sono al termine del ciclo di vita del supporto. Per ulteriori informazioni sul ciclo di vita dei prodotti, visitare il sito Web [Ciclo di vita del supporto Microsoft](http://support.microsoft.com/common/international.aspx?rdpath=gp;%5Bln%5D;lifecycle).

Per evitare di esporre i propri sistemi a potenziali vulnerabilità, si raccomanda ai clienti che utilizzano versioni precedenti del software di eseguire quanto prima la migrazione a versioni supportate. Per informazioni sulla disponibilità del supporto per la versione del software in uso, vedere [Selezionare un prodotto per informazioni sul ciclo di vita](http://support.microsoft.com/gp/lifeselect). Per ulteriori informazioni sui service pack relativi a queste versioni del software, vedere [Criteri relativi al ciclo di vita del supporto per i service pack](http://support.microsoft.com/gp/lifesupsps).

I clienti che hanno ancora la necessità di servizi di supporto per versioni precedenti del software sono invitati a contattare il loro rappresentante del team Microsoft, il responsabile tecnico dei rapporti con i clienti oppure il rappresentante del partner Microsoft appropriato per informazioni sui servizi di supporto personalizzati. I clienti che non dispongono di un contratto Alliance, Premier o Authorized possono contattare le filiali Microsoft locali. Per informazioni sui contatti, visitare il sito Web [Microsoft Worldwide Information](http://www.microsoft.com/worldwide/), selezionare il Paese desiderato dall'elenco Informazioni sui contatti e fare clic su **Go** per visualizzare un elenco di numeri telefonici. Quando si effettua la chiamata, richiedere di parlare con il responsabile locale delle vendite per i servizi di supporto Premier. Per ulteriori informazioni, vedere le [Domande frequenti sui criteri di ciclo di vita del supporto Microsoft](http://support.microsoft.com/gp/lifepolicy).

Vulnerabilità ad attacchi di tipo Denial of Service dovuta a collisioni in HashTable - CVE-2011-3414
----------------------------------------------------------------------------------------------------

<span></span>
È stata riscontrata una vulnerabilità ad attacchi di tipo Denial of Service nel modo in cui ASP.NET Framework gestisce richieste appositamente predisposte, causando una collisione hash. Un utente malintenzionato in grado di sfruttare questa vulnerabilità può inviare un numero ridotto di richieste appositamente predisposte a un server ASP.NET, causando un degrado delle prestazioni sufficientemente forte da causare una condizione di tipo Denial of Service.

Per visualizzare questa vulnerabilità come voce standard nell'elenco CVE, vedere il codice [CVE-2011-3414](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3414).

#### Fattori attenuanti per la vulnerabilità ad attacchi di tipo Denial of Service dovuta a collisioni in HashTable - CVE-2011-3414

Per fattore attenuante si intende un'impostazione, una configurazione comune o una procedura consigliata generica esistente in uno stato predefinito in grado di ridurre la gravità nello sfruttamento di una vulnerabilità. I seguenti fattori attenuanti possono essere utili per l'utente:

-   Per impostazione predefinita, IIS non è attivato in nessun sistema operativo Windows
-   I siti che rifiutano i tipi di contenuto HTTP application/x-www-form-urlencoded o multipart/form-data non sono vulnerabili
-   Per impostazione predefinita, ASP.NET non viene installato quando si installa .NET Framework. Sono pertanto esposti a questa vulnerabilità solo i clienti che installano e attivano ASP.NET manualmente.

#### Soluzioni alternative per la vulnerabilità ad attacchi di tipo Denial of Service dovuta a collisioni in HashTable - CVE-2011-3414

Per soluzione alternativa si intende un'impostazione o una modifica nella configurazione che non elimina la vulnerabilità sottostante, ma consente di bloccare gli attacchi noti prima di applicare l'aggiornamento. Microsoft ha verificato le seguenti soluzioni alternative e segnala nel corso della discussione se tali soluzioni riducono la funzionalità:

-   **Soluzione alternativa basata sulla configurazione**

    La soluzione alternativa seguente configura il limite della dimensione massima della richiesta che ASP.NET accetta da un client. Diminuendo la dimensione massima della richiesta si diminuisce la vulnerabilità del server ASP.NET a un attacco di tipo Denial of Service.

    **Nota:** eseguire il back up della configurazione in uso prima di apportarvi modifiche.

    Questo valore di configurazione può essere applicato in modo globale a tutti i siti ASP.NET in un server aggiungendo la voce a web.config o applicationhost.config principali. In alternativa, questa configurazione può essere limitata a un sito o a un'applicazione particolari aggiungendola a un file web.config per quel sito o quell'applicazione. Per ulteriori informazioni sulla configurazione di ASP.NET, fare riferimento all'articolo MSDN [Cenni preliminari sulla configurazione di ASP.NET](http://msdn.microsoft.com/it-it/library/ms178683.aspx).

    1.  Se l'applicazione utilizza ViewState, aggiungere la configurazione seguente al file di configurazione ASP.NET appropriato per limitare a 200 KB la dimensione massima di richiesta che ASP.NET accetterà da un client.

        `<configuration><system.web><httpRuntime maxRequestLength="200”/></system.web></configuration>`

    2.  Se l'applicazione non utilizza ASP.NET ViewState, aggiungere la configurazione seguente al file di configurazione ASP.NET appropriato per limitare a 20 KB la dimensione massima di richiesta che ASP.NET accetterà da un client.

        `<configuration><system.web><httpRuntime maxRequestLength="20”/></system.web></configuration>`

    Per ulteriori informazioni sull'opzione di configurazione maxRequestLength, fare riferimento alla documentazione MSDN sull'[Elemento httpRuntime](http://msdn.microsoft.com/it-it/library/e1f13641.aspx).

    **Nota:** L'aumento dei limiti predefiniti della stringa di query HTTP e delle dimensioni dell'intestazione della richiesta aumenta la vulnerabilità del server al problema di tipo Denial of Service descritto in questo advisory. Per ulteriori informazioni su come impostare questi limiti, vedere [Impostazioni del Registro di sistema di Http.sys per IIS](http://support.microsoft.com/kb/820129).

    **Impatto della soluzione alternativa:** Le richieste del client che superano il valore maxRequestLength configurato hanno come risultato un ConfigurationErrorsException sul lato server e uno stato di errore http inviato al client. I limiti configurati in precedenza devono essere utilizzati come una linea di base e possono essere regolati per evitare problemi nell'applicazione. La configurazione del valore di maxRequestLength su un valore più piccolo riduce anche la vulnerabilità del server a questo particolare attacco di tipo Denial of Service. Le applicazioni che accettano caricamenti file di client e quelle che generano un grande viewstate sono esempi di applicazioni che possono essere interessate da questa soluzione alternativa.

    Dopo aver applicato questa soluzione alternativa, nel Registro eventi applicazioni potrebbe essere visualizzata la seguente mancata riuscita di una richiesta client.

    `Codice evento: 3004 Messaggio dell'evento: Limite delle dimensioni di invio superato. Informazioni eccezione:     Tipo eccezione: HttpException     Messaggio eccezione: Superata la lunghezza massima della richiesta.   in System.Web.HttpRequest.GetEntireRawContent()   in System.Web.HttpRequest.FillInFormCollection()   in System.Web.HttpRequest.get_Form()   in System.Web.HttpRequest.get_HasForm()   in System.Web.UI.Page.GetCollectionBasedOnMethod(Boolean dontReturnNull)   in System.Web.UI.Page.DeterminePostBackMode()   in System.Web.UI.Page.ProcessRequestMain(Boolean includeStagesBeforeAsyncPoint, Boolean includeStagesAfterAsyncPoint)`

    In alternativa, la mancata riuscita della richiesta può produrre la voce seguente in un file di registro IIS.

    `#Fields: s-ip cs-method cs-uri-stem cs-uri-query s-port cs-username c-ip cs(User-Agent) sc-status sc-substatus sc-win32-status time-taken 127.0.0.1 POST /webSite/Default.aspx - 80 - 127.0.0.1 - 500 0 0 1268`

    **Rimozione della soluzione alternativa:** Per annullare la soluzione alternativa, rimuovere le sezioni della configurazione che erano state aggiunte in questa soluzione alternativa o tornare a un backup della configurazione prima dell'applicazione delle soluzioni alternative.

#### Domande frequenti per la vulnerabilità ad attacchi di tipo Denial of Service dovuta a collisioni in HashTable - CVE-2011-3414

**Qual è la portata o l'impatto di questa vulnerabilità?**
Si tratta di una vulnerabilità ad attacchi di tipo Denial of Service. Questa vulnerabilità consente a un utente malintenzionato non autenticato di ridurre le prestazioni di un sito ASP.NET creando una condizione di Denial of Service.

**Quali sono le cause di questa vulnerabilità?**
La vulnerabilità è causata dal modo in cui ASP.NET esegue l'hashing delle richieste appositamente predisposte e inserisce tali dati in una tabella hash, causando una collisione hash. Quando molte di queste collisioni sono concatenate, le prestazioni della tabella hash sono considerevolmente ridotte, causando una condizione di Denial of Service.

**A quali attacchi viene esposto il sistema a causa di questa vulnerabilità?**
Un utente malintenzionato può utilizzare questa vulnerabilità per causare un attacco di tipo Denial of Service e impedire la disponibilità di siti che utilizzano ASP.NET.

**In quale modo un utente malintenzionato può sfruttare questa vulnerabilità?**
Un utente malintenzionato non autenticato può inviare un numero limitato di richieste ASP.NET appositamente predisposte a un sito ASP.NET interessato, causando una condizione di tipo Denial of Service.

**Quali sono i sistemi principalmente interessati da questa vulnerabilità?**
I sistemi per Internet con ASP.NET installato sono maggiormente interessati da questa vulnerabilità. Inoltre, i siti Web interni che utilizzano l'autenticazione dei moduli ASP.NET possono essere interessati da questa vulnerabilità.

**Quali sono gli scopi dell'aggiornamento?**
L'aggiornamento risolve questa vulnerabilità correggendo come il modo in cui .NET Framework gestisce le richieste appositamente predisposte.

**Al momento del rilascio di questo bollettino, le informazioni sulla vulnerabilità erano disponibili pubblicamente?**
Sì. Le informazioni sulla vulnerabilità sono state divulgate pubblicamente. Questa vulnerabilità è stata identificata con il codice [CVE-2011-3414](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3414).

**Al momento del rilascio di questo bollettino, erano già stati segnalati attacchi basati sullo sfruttamento di questa vulnerabilità?**
No. Al momento della pubblicazione del presente bollettino, Microsoft non aveva ricevuto alcuna segnalazione in merito allo sfruttamento di questa vulnerabilità a scopo di attacco.

**Si tratta di un problema del settore?**
Gli attacchi che tentano di utilizzare questo tipo di vulnerabilità sono genericamente conosciuti come attacchi di collisione hash. Gli attacchi di questo tipo non sono specifici delle tecnologie Microsoft ed interessano altri fornitori di software di servizi Web. Si consiglia di contattare il proprio fornitore di piattaforma Web per informazioni o aggiornamenti.

Il ticket di autenticazione dei moduli è contenuto di solito in un cookie. Tuttavia, ASP.NET versione 2.0 supporta l'autenticazione di moduli senza cookie, causando il passaggio del ticket a una stringa di query. L'utente malintenzionato può quindi intraprendere un'azione qualsiasi nel contesto della vittima, compresa l'esecuzione di comandi non autorizzati sul sito.

**Quali sono i sistemi principalmente interessati da questa vulnerabilità?**
I sistemi per Internet con ASP.NET installato sono maggiormente interessati da questa vulnerabilità. Inoltre, i siti Web interni che utilizzano l'autenticazione dei moduli ASP.NET possono essere interessati da questa vulnerabilità.

**Se un sito Web utilizza una Web farm, è necessario aggiornare tutti i** **computer** **nella Web farm?**
Sì. Questo aggiornamento modifica il formato del ticket di autenticazione dei moduli in modo incompatibile con il formato precedente. Questo significa che i ticket generati utilizzando il nuovo comportamento non possono essere letti dai computer che utilizzano il vecchio comportamento e viceversa. Gli amministratori le cui applicazioni utilizzano l'autenticazione basata su moduli devono intraprendere azioni specifiche quando implementano questo aggiornamento per la protezione per assicurare che tutti i loro server passino al nuovo comportamento contemporaneamente.

Per ulteriori informazioni, vedere l'[articolo di Microsoft TechNet 2659968](http://support.microsoft.com/kb/2659968).

**È possibile aggiornare i** **computer** **nella Web farm uno alla volta** **anzichétutti contemporaneamente?**
Sì. Se non è possibile implementare MS11-100 su tutti i server in una determinata Web farm simultaneamente, è possibile impostare un'opzione di compatibilità nel file web.config o machine.config prima di installare l'aggiornamento per forzare il vecchio comportamento dopo che l'aggiornamento è stato installato. L'opzione di compatibilità da utilizzare è la seguente:

`<appSettings><add key="aspnet:UseLegacyFormsAuthenticationTicketCompatibility" value="true" /></appSettings>`

L'impostazione di questa opzione consentirà alle Web farm in cui solo alcuni computer sono stati aggiornati di continuare a funzionare correttamente. Tuttavia, è importante tenere presente che i computer con l'opzione di configurazione impostata si troveranno in uno stato non sicuro e non si avvarranno della soluzione contenuta in questo aggiornamento per la protezione. Dopo che MS11-100 è stato distribuito in tutti i computer in una farm, l'opzione di configurazione deve essere rimossa per attivare il nuovo comportamento sicuro.

**Questo aggiornamento rispetterà l'impostazione ticketCompatibilityMode dell'autenticazione dei moduli del sito in uso?**
No. Poiché la soluzione contenuta in questo aggiornamento per la protezione modifica il formato dei ticket di autenticazione dei moduli, l'impostazione della configurazione ticketCompatibilityMode non è più supportata dopo l'installazione di MS11-100.

**Quali sono gli scopi dell'aggiornamento?**
L'aggiornamento risolve questa vulnerabilità correggendo il modo in cui ASP.NET Framework autentica gli utenti.

No. Al momento della pubblicazione del presente bollettino, Microsoft non aveva ricevuto alcuna segnalazione in merito allo sfruttamento di questa vulnerabilità a scopo di attacco.

Vulnerabilità legata alla memorizzazione nella cache dei ticket di autenticazione dei moduli ASP.NET - CVE-2011-3417
--------------------------------------------------------------------------------------------------------------------

<span></span>
È stata riscontrata una vulnerabilità legata all'acquisizione di privilegi più elevati nel modo in cui ASP.NET Framework gestisce i contenuti memorizzati nella cache quando si utilizza l'autenticazione dei moduli con data di scadenza variabile. Un utente malintenzionato in grado di sfruttare questa vulnerabilità può intraprendere qualsiasi azione che l'utente può intraprendere sul sito nel contesto della vittima, compresa l'esecuzione di comandi non autorizzati. In uno scenario di attacco tramite posta elettronica, l'utente malintenzionato può sfruttare la vulnerabilità inviando un collegamento appositamente predisposto all'utente e inducendo quest'ultimo ad aprirlo. Non è in alcun modo possibile obbligare gli utenti a visitare un sito Web. L'utente malintenzionato deve invece convincere gli utenti a compiere un'azione, solitamente facendo clic su un collegamento in un messaggio di posta elettronica o in Instant Messenger.

Per visualizzare questa vulnerabilità come voce standard nell'elenco CVE, vedere il codice [CVE-2011-3417](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3417).

#### Fattori attenuanti per la vulnerabilità legata alla memorizzazione nella cache dei ticket di autenticazione dei moduli ASP.NET - CVE-2011-3417

Per fattore attenuante si intende un'impostazione, una configurazione comune o una procedura consigliata generica esistente in uno stato predefinito in grado di ridurre la gravità nello sfruttamento di una vulnerabilità. I seguenti fattori attenuanti possono essere utili per l'utente:

-   Per impostazione predefinita, le risposte ASP.NET non vengono memorizzate nella cache da OutputCache. Lo sviluppatore del sito deve abilitare la cache di output tramite la direttiva OutputCache su una pagina.
-   Sfruttando questa vulnerabilità, un utente malintenzionato può ottenere gli stessi diritti utente della vittima. Pertanto, gli utenti con account configurati in modo da disporre solo di diritti limitati sono esposti all'attacco in misura inferiore rispetto a quelli che operano con privilegi di amministrazione.
-   Per impostazione predefinita, IIS non è installato in nessuna versione dei sistemi operativi interessati. Sono pertanto esposti a questa vulnerabilità solo i clienti che installano questo servizio manualmente.
-   Per impostazione predefinita, ASP.NET non viene installato quando si installa .NET. Sono pertanto esposti a questa vulnerabilità solo i clienti che installano e attivano ASP.NET manualmente.

#### Soluzioni alternative per la vulnerabilità legata alla memorizzazione nella cache dei ticket di autenticazione dei moduli ASP.NET - CVE-2011-3417

Per soluzione alternativa si intende un'impostazione o una modifica nella configurazione che non elimina la vulnerabilità sottostante, ma consente di bloccare gli attacchi noti prima di applicare l'aggiornamento. Microsoft ha verificato le seguenti soluzioni alternative e segnala nel corso della discussione se tali soluzioni riducono la funzionalità:

-   **Limitare i cookie di autenticazione dei moduli ai canali SSL**

    Aprire l'applicazione o il file web.config globale dopo aver eseguito una copia di backup.

    Impostare requireSSL="true" nell'elemento &lt;forms&gt;, come mostrato nel codice seguente.

    `<forms loginUrl="Secure\Login.aspx"    requireSSL="true" />`

    **Per annullare il risultato della soluzione alternativa.**

    Ripristinare il file web.config dalla copia di backup.

-   **Disattivare la scadenza variabile per i cookie di autenticazione dei moduli**

    1.  Aprire l'applicazione o il file web.config globale dopo aver eseguito una copia di backup
    2.  Impostare requireSSL="false" nell'elemento &lt;forms&gt;, come mostrato nel codice seguente.

        `<forms slidingExpiration="false"/>`

    **Impatto della soluzione alternativa.** Gli utenti devono eseguire nuovamente l'accesso alla scadenza dei ticket.

    **Per annullare il risultato della soluzione alternativa.**

    Ripristinare il file web.config dalla copia di backup.

-   **Disattivare OutputCache**

    1.  Eseguire il backup della cartella dell'applicazione Web.
    2.  Rimuovere tutte le direttive &lt;%@ OutputCache %&gt; nelle pagine Web.
    3.  Se il sito web è un'applicazione MVC, rimuovere anche tutti gli attributi OutputCache dalle azioni del controller.

    **Impatto della soluzione alternativa.** Gli utenti devono eseguire nuovamente l'accesso alla scadenza dei ticket.

    **Per annullare il risultato della soluzione alternativa.**

    Ripristinare la cartella dell'applicazione Web.

#### Domande frequenti per la vulnerabilità legata alla memorizzazione nella cache dei ticket di autenticazione dei moduli ASP.NET - CVE-2011-3417

**Qual è la portata o l'impatto di questa vulnerabilità?**
Questa è una vulnerabilità legata all'acquisizione di privilegi più elevati tramite la quale un utente malintenzionato remoto non autenticato accede a un altro account sull'applicazione .NET.

**Quali sono le cause di questa vulnerabilità?**
Questa vulnerabilità si verifica quando .NET Framework gestisce erroneamente i contenuti memorizzati nella cache quando si utilizza l'autenticazione dei moduli con data di scadenza variabile.

**A quali attacchi viene esposto il sistema a causa di questa vulnerabilità?**
Un utente malintenzionato in grado di sfruttare questa vulnerabilità può intraprendere qualsiasi azione che l'utente può intraprendere sul sito nel contesto della vittima, compresa l'esecuzione di comandi non autorizzati.

**In quale modo un utente malintenzionato può sfruttare questa vulnerabilità?**
In uno scenario di attacco tramite posta elettronica, l'utente malintenzionato può sfruttare la vulnerabilità inviando un collegamento appositamente predisposto all'utente e inducendo quest'ultimo ad aprirlo. Non è in alcun modo possibile obbligare gli utenti a visitare un sito Web. L'utente malintenzionato deve invece convincere gli utenti a compiere un'azione, solitamente facendo clic su un collegamento in un messaggio di posta elettronica o in Instant Messenger.

**Che cos'è l'autenticazione dei moduli in ASP.NET?**
L'autenticazione dei moduli utilizza un ticket di autenticazione creato quando un utente accede a un sito, quindi segue l'utente in tutto il sito. Il ticket di autenticazione dei moduli è contenuto di solito in un cookie. Tuttavia, ASP.NET versione 2.0 supporta l'autenticazione di moduli senza cookie, causando il passaggio del ticket a una stringa di query. Per ulteriori informazioni, consultare l'articolo di MSDN [Spiegazione: autenticazione dei moduli in ASP.NET 2.0](http://msdn.microsoft.com/en-us/library/ff647070.aspx).

**Quali sono i sistemi principalmente interessati da questa vulnerabilità?**
I sistemi per Internet con IIS e ASP.NET installati sono maggiormente interessati da questa vulnerabilità. Inoltre, i siti Web interni che utilizzano l'autenticazione dei moduli ASP.NET possono essere interessati da questa vulnerabilità.

**Quali sono gli scopi dell'aggiornamento?**
L'aggiornamento risolve il modo in cui ASP.NET gestisce il contenuto memorizzato nella cache.

#### Informazioni per la distribuzione

**Installazione dell'aggiornamento**

Quando si installa questo aggiornamento per la protezione, il programma di installazione verifica se uno o più file da aggiornare nel sistema sono stati aggiornati in precedenza mediante una correzione rapida Microsoft.

Se uno di questi file è stato aggiornato mediante l'installazione di una correzione rapida precedente, il programma di installazione copia nel sistema i file della cartella RTMQFE, SP1QFE o SP2QFE. In alternativa, copia i file RTMGDR, SP1GDR o SP2GDR. Gli aggiornamenti per la protezione potrebbero non contenere tutte le variazioni di questi file. Per ulteriori informazioni in merito, vedere l'[articolo della Microsoft Knowledge Base 824994](http://support.microsoft.com/kb/824994).

Per ulteriori informazioni sul programma di installazione, vedere l'[articolo della Microsoft Knowledge Base 832475](http://support.microsoft.com/kb/832475).

Per ulteriori informazioni sulla terminologia utilizzata nel presente bollettino, ad esempio su ciò che si intende per *correzione rapida*, vedere l'[articolo della Microsoft Knowledge Base 824684](http://support.microsoft.com/kb/824684/it).

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi su **Cerca**.
    2.  Nel riquadro **Risultati ricerca**, scegliere **Tutti i file e le cartelle** in **Ricerca guidata**.
    3.  Nella casella **Nome del file o parte del nome** , digitare il nome del file, individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Cerca**.
    4.  Nell'elenco dei file, fare clic con il pulsante destro del mouse sul file individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Proprietà**.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.
    5.  Nella scheda **Versione** individuare il numero di versione del file installato nel sistema e confrontarlo con quello specificato nell'appropriata tabella di informazioni sui file.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.

Inoltre, se un amministratore o un OEM integra o esegue lo slipstreaming dell'aggiornamento per la protezione nei file di origine dell'installazione di Windows, è possibile che queste chiavi del Registro di sistema non vengano create correttamente.

Quando si installa questo aggiornamento per la protezione, il programma di installazione verifica se uno o più file da aggiornare nel sistema sono stati aggiornati in precedenza mediante una correzione rapida Microsoft.

**Verifica dell'applicazione dell'aggiornamento**

-   **Microsoft Baseline Security Analyzer**

    Per verificare se un aggiornamento per la protezione è stato applicato al sistema interessato, è possibile utilizzare lo strumento Microsoft Baseline Security Analyzer (MBSA). Per ulteriori informazioni, vedere la sezione **Strumenti e informazioni sul rilevamento e sulla distribuzione** nel presente bollettino.

-   **Verifica della versione dei file**

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi su **Cerca**.
    2.  Nel riquadro **Risultati ricerca**, scegliere **Tutti i file e le cartelle** in **Ricerca guidata**.
    3.  Nella casella **Nome del file o parte del nome** , digitare il nome del file, individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Cerca**.
    4.  Nell'elenco dei file, fare clic con il pulsante destro del mouse sul file individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Proprietà**.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.

**Verifica dell'applicazione dell'aggiornamento**

    Per verificare se un aggiornamento per la protezione è stato applicato al sistema interessato, è possibile utilizzare lo strumento Microsoft Baseline Security Analyzer (MBSA). Per ulteriori informazioni, vedere la sezione **Strumenti e informazioni sul rilevamento e sulla distribuzione** nel presente bollettino.

-   **Verifica della versione dei file**

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi immettere il nome di un file aggiornato nella casella **Inizia ricerca**.
    2.  Quando il file viene visualizzato in **Programmi**, fare clic con il pulsante destro del mouse sul nome del file, quindi scegliere **Proprietà**.
    3.  Nella scheda **Generale**, confrontare le dimensioni del file con le tabelle di informazioni sui file fornite nell'articolo della Knowledge Base indicato nel bollettino.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.
    4.  È possibile inoltre selezionare la scheda **Dettagli** e confrontare le informazioni (ad esempio la versione e la data di modifica del file) con le tabelle di informazioni sui file fornite nell'articolo della Knowledge Base indicato nel bollettino.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.
    5.  Un'ultima possibilità è fare clic sulla scheda **Versioni precedenti** e confrontare le informazioni relative alla versione precedente del file con quelle della versione nuova o aggiornata.

**Verifica dell'applicazione dell'aggiornamento**

-   **Microsoft Baseline Security Analyzer**

    Per verificare se un aggiornamento per la protezione è stato applicato al sistema interessato, è possibile utilizzare lo strumento Microsoft Baseline Security Analyzer (MBSA). Per ulteriori informazioni, vedere la sezione **Strumenti e informazioni sul rilevamento e sulla distribuzione** nel presente bollettino.

-   **Verifica della versione dei file**

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi immettere il nome di un file aggiornato nella casella **Inizia ricerca**.
    2.  Quando il file viene visualizzato in **Programmi**, fare clic con il pulsante destro del mouse sul nome del file, quindi scegliere **Proprietà**.
    3.  Nella scheda **Generale**, confrontare le dimensioni del file con le tabelle di informazioni sui file fornite nell'articolo della Knowledge Base indicato nel bollettino.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.
    4.  È possibile inoltre selezionare la scheda **Dettagli** e confrontare le informazioni (ad esempio la versione e la data di modifica del file) con le tabelle di informazioni sui file fornite nell'articolo della Knowledge Base indicato nel bollettino.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.
    5.  Un'ultima possibilità è fare clic sulla scheda **Versioni precedenti** e confrontare le informazioni relative alla versione precedente del file con quelle della versione nuova o aggiornata.

| Opzione                                 | Descrizione                                                                                                                                                                                      |
|-----------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|

| **/norestart**                          | Se utilizzato in combinazione con **/quiet**, consente di non riavviare il sistema dopo l'installazione anche nel caso in cui il riavvio sia necessario per il completamento dell'installazione. |
