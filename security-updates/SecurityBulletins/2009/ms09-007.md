Security Bulletin

Bollettino Microsoft sulla sicurezza MS09-007 - Importante
==========================================================

Una vulnerabilità presente in SChannel può consentire attacchi di spoofing (960225)
-----------------------------------------------------------------------------------

Data di pubblicazione: martedì 10 marzo 2009

**Versione:** 1.0

### Informazioni generali

#### Riepilogo

Questo aggiornamento per la protezione risolve una vulnerabilità del pacchetto di protezione Canale sicuro (SChannel) di Windows, che è stata segnalata privatamente. Tale vulnerabilità può consentire attacchi di spoofing se un utente malintenzionato ottiene l'accesso al certificato utilizzato da un utente finale per l'autenticazione. I clienti sono interessati da questa vulnerabilità solo se il componente chiave pubblico del certificato utilizzato per l'autenticazione viene intercettato dall'utente malintenzionato con altri mezzi.

Questo aggiornamento per la protezione è considerato di livello importante per tutte le edizioni supportate di Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista e Windows Server 2008. Per ulteriori informazioni, vedere la sottosezione **Software interessato e Software non interessato** all'interno della presente sezione.

L'aggiornamento per la protezione risolve la vulnerabilità modificando il modo in cui il server analizza i dati di scambio chiave durante l'handshake TLS. Per ulteriori informazioni sulla vulnerabilità, vedere la sottosezione Domande frequenti per la specifica voce della vulnerabilità nella sezione **Informazioni sulle vulnerabilità**.

**Raccomandazione.** Se la funzionalità Aggiornamenti automatici è abilitata, gli utenti non devono intraprendere alcuna azione, poiché questo aggiornamento per la protezione viene scaricato e installato automaticamente. Gli utenti che non hanno attivato la funzionalità Aggiornamenti automatici devono verificare la disponibilità di aggiornamenti e installare questo aggiornamento manualmente. Per informazioni sulle opzioni di configurazione specifiche relative agli aggiornamenti automatici, vedere l'[articolo della Microsoft Knowledge Base 294781](http://support.microsoft.com/kb/294871).

Per gli amministratori e le installazioni delle organizzazioni o gli utenti finali che desiderano installare manualmente questo aggiornamento per la protezione, Microsoft consiglia di applicare l'aggiornamento quanto prima utilizzando il software di gestione degli aggiornamenti o verificando la disponibilità degli aggiornamenti tramite il servizio [Microsoft Update](http://update.microsoft.com/microsoftupdate/v6/default.aspx?ln=it-it).

**Problemi noti.** Nessuno.

Vulnerabilità legata ad attacchi di spoofing in SChannel - CVE-2009-0085  
------------------------------------------------------------------------
  
<span></span>
Una vulnerabilità legata ad attacchi di spoofing è stata riscontrata nel componente di autenticazione di Microsoft Windows SChannel, quando viene utilizzata l'autenticazione basata sui certificati. Sfruttando questa vulnerabilità, un utente malintenzionato può eseguire l'autenticazione presso un server utilizzando solamente il certificato digitale di un utente autorizzato, senza disporre della chiave privata associata.
  
Per visualizzare questa vulnerabilità come voce standard nell'elenco CVE, vedere il codice [CVE-2009-0085](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0085).
  
#### Fattori attenuanti della vulnerabilità legata ad attacchi di spoofing in SChannel - CVE-2009-0085
  
Per fattore attenuante si intende un'impostazione, una configurazione comune o una procedura consigliata generica esistente in uno stato predefinito in grado di ridurre la gravità nello sfruttamento di una vulnerabilità. I seguenti fattori attenuanti possono essere utili per l'utente:
  
-   I clienti sono interessati solo se il certificato utilizzato per l'autenticazione è già stato intercettato dall'utente malintenzionato con altri mezzi. Nel caso di un server Web IIS, i certificati non vengono trasferiti in forma di testo non crittografato durante il processo di autenticazione.  
-   Solo gli account utente configurati localmente risultano perciò vulnerabili. I clienti che eseguono il mapping dei certificati degli utenti autorizzati delle loro applicazioni su un dominio Active Directory non sono interessati da questa vulnerabilità.
  
#### Soluzioni alternative per la vulnerabilità legata ad attacchi di spoofing in SChannel - CVE-2009-0085
  
Per soluzione alternativa si intende un'impostazione o una modifica nella configurazione che non elimina la vulnerabilità sottostante, ma consente di bloccare gli attacchi noti prima di applicare l'aggiornamento. Microsoft ha verificato le seguenti soluzioni alternative e segnala nel corso della discussione se tali soluzioni riducono la funzionalità:
  
-   I clienti possono implementare il mapping dei certificati di Active Directory, che non è interessato da questa vulnerabilità. Il mapping dei certificati di Active Directory consente a un utente che disponga di una chiave pubblica attendibile di accedere alle risorse di dominio senza digitare un nome utente e una password. È possibile eseguire il mapping di un certificato client direttamente su un account utente di Active Directory.  
    Per ulteriori informazioni sui diversi modi di configurare il mapping dei certificati, vedere [Map certificates to user accounts](http://technet.microsoft.com/en-us/library/cc736706.aspx).
  
#### Domande frequenti sulla vulnerabilità legata ad attacchi di spoofing in SChannel - CVE-2009-0085
  
**Qual è la portata o l'impatto di questa vulnerabilità?**    
Una vulnerabilità legata ad attacchi di spoofing è stata riscontrata nel componente di autenticazione di Microsoft Windows SChannel, quando viene utilizzata l'autenticazione basata sui certificati. Sfruttando questa vulnerabilità, un utente malintenzionato può agire per conto di un altro utente ed eseguire l'autenticazione su un server protetto utilizzando il componente pubblico della credenziale di autenticazione dell'utente stesso.
  
**Quali sono le cause di questa vulnerabilità?**    
Il componente di autenticazione di SChannel non convalida in modo sufficiente determinati messaggi di handshake di TLS (Transport Layer Security) al fine di garantire che il client abbia realmente accesso alla chiave privata collegata al certificato utilizzato per l'autenticazione.
  
**Che cos'è il pacchetto di protezione Canale sicuro (Schannel)?**    
Il pacchetto di protezione Canale sicuro (SChannel) è un SSP (Security Support Provider) che implementa i protocolli di autenticazione Internet standard SSL (Secure Sockets Layer) e TLS (Transport Layer Security). Questi componenti vengono utilizzati per garantire delle comunicazioni protette tra diverse applicazioni comuni in Internet e in rete, come ad esempio i browser Web. SChannel fa parte del pacchetto di protezione che fornisce un servizio di autenticazione in modo da garantire comunicazioni protette tra client e server. Per ulteriori informazioni su SChannel, visitare il sito Web di [SChannel](http://msdn.microsoft.com/en-us/library/ms678421.aspx) su MSDN.
  
**Che cos'è un certificato digitale x.509?**    
Le tecniche di crittografia che utilizzano chiavi pubbliche e private necessitano di una infrastruttura a chiave pubblica (PKI, Public Key Infrastructure) per poter supportare la distribuzione e la certificazione delle chiavi pubbliche. I certificati digitali contengono chiavi pubbliche, informazioni sugli algoritmi utilizzati, dati relativi al proprietario o all'oggetto, la firma digitale dell'Autorità di certificazione che ha verificato i dati relativi all'oggetto e l'indicazione di un intervallo di tempo in cui il certificato sia da considerarsi valido.
  
I certificati digitali possono essere convalidati da un client come Internet Explorer per assicurare che il server al quale un utente si connette sia di fatto il server autorizzato per quel nome host. I certificati possono tuttavia essere utilizzati anche come una credenziale di protezione per l'autenticazione e l'autorizzazione degli utenti.
  
**Che cos'è la libreria Microsoft Secure Sockets Layer?**    
La libreria Microsoft Secure Sockets Layer contiene il supporto per diversi protocolli di comunicazione protetti tra cui Transport Layer Security 1.0 (TLS 1.0), Secure Sockets Layer 3.0 (SSL 3.0), l'obsoleto e raramente utilizzato Secure Sockets Layer 2.0 (SSL 2.0) e Private Communication Technology 1.0 (PCT 1.0).
  
Questi protocolli garantiscono una connessione crittografata tra server e client. SSL è in grado di proteggere le informazioni quando gli utenti si connettono attraverso reti pubbliche come Internet. Il supporto SSL richiede un certificato SSL, che deve essere installato su un server. Per ulteriori informazioni su SSL, vedere [XADM: come funziona Secure Sockets Layer](http://support.microsoft.com/default.aspx?scid=kb;en-us;q245152).
  
**Che cos'è un handshake TLS?**    
Il protocollo di handshake di Transport Layer Security (TLS) è responsabile dell'autenticazione e dello scambio delle chiavi, vale a dire di due operazioni necessarie per stabilire o ripristinare delle sessioni protette. Quando si stabilisce una sessione protetta, il protocollo di handshake gestisce le seguenti operazioni:
  
-   negoziazione di pacchetti di crittografia  
-   autenticazione del server e facoltativamente del client  
-   scambio di informazioni relative alla chiave di sessione
  
**Che cos'è un SSP (Security Support Provider)?**    
Un provider supporto Protezione implementa un'interfaccia comune tra applicazioni a livello di trasporto, come Microsoft RPC (Remote Procedure Call), e provider di protezione, come Windows Distributed Security. Per fare ciò, crea uno o più pacchetti di protezione disponibili per le applicazioni. Ogni pacchetto di protezione fornisce dei mapping che abbinano le chiamate di funzione dell'interfaccia SSP dell'applicazione alle funzioni di un protocollo di protezione effettivo. I pacchetti di protezione supportano protocolli di protezione quali l'autenticazione di Kerberos e Microsoft LAN Manager.
  
**A quali attacchi viene esposto il sistema a causa di questa vulnerabilità?**    
Sfruttando questa vulnerabilità, un utente malintenzionato può eseguire l'autenticazione su un server protetto e agire per conto di un utente autorizzato, pur non avendo accesso alla chiave privata dell'utente stesso. Tale chiave viene normalmente richiesta al fine di eseguire una corretta autenticazione, quando il server è stato configurato in modo da richiedere l'autenticazione del client.
  
**In quale modo un utente malintenzionato può sfruttare questa vulnerabilità?**    
Un utente malintenzionato deve ottenere l'accesso al componente pubblico del certificato effettivo utilizzato dall'utente finale per l'autenticazione. Egli può in seguito predisporre un pacchetto TLS (Transport Layer Security) per eludere la convalida applicata dal componente di SChannel per accertare che la chiave privata sia realmente presente durante l'autenticazione.
  
Nelle distribuzioni più comuni, i certificati utilizzati per l'autenticazione del client sono creati per un unico scopo, configurato in un client di Windows. Tali certificati vengono trasferiti per uno scopo specifico, come quello dell'autenticazione su Web. Nel caso di una connessione SSL a un server Web IIS (Internet Information Services), la chiave pubblica non viene mai trasferita in formato testo non crittografato. Tuttavia, in casi in cui un unico certificato venga utilizzato simultaneamente per autorizzare il client e per altri scopi, come nel caso della firma di un messaggio di posta elettronica, o nei casi in cui il certificato venga utilizzato per autenticare un utente presente in diversi siti Web, la chiave pubblica può subire una distribuzione più ampia.
  
**Quali sono i sistemi principalmente interessati da questa vulnerabilità?**    
La vulnerabilità interessa servizi come i server Web che utilizzano l'autenticazione basata sui certificati e che richiedono all'utente di provare il possesso della chiave privata relativa al certificato durante il processo di autenticazione.
  
**Quali sono gli scopi dell'aggiornamento?**    
L'aggiornamento risolve la vulnerabilità modificando il modo in cui il server analizza i dati di scambio chiave durante l'handshake TLS.
  
**Al momento del rilascio di questo bollettino le informazioni sulla vulnerabilità erano disponibili pubblicamente?**    
No. Microsoft ha ricevuto informazioni sulla vulnerabilità da fonti private. Secondo le informazioni in possesso di Microsoft, al momento della pubblicazione del presente bollettino l'esistenza di questa vulnerabilità non era stata ancora divulgata pubblicamente. Il presente bollettino sulla sicurezza risolve questa vulnerabilità nonché altri problemi scoperti tramite ricerche interne.
  
**Al momento del rilascio di questo bollettino erano già stati segnalati attacchi basati sullo sfruttamento di questa vulnerabilità?**    
No. Al momento della pubblicazione del presente bollettino, Microsoft non aveva ricevuto alcuna segnalazione in merito allo sfruttamento di questa vulnerabilità a scopo di attacco, né dell'esistenza di un codice di prova pubblicato.

**Verifica dell'applicazione dell'aggiornamento**

-   **Microsoft Baseline Security Analyzer**

    Per verificare se un aggiornamento per la protezione è stato applicato al sistema interessato, è possibile utilizzare lo strumento Microsoft Baseline Security Analyzer (MBSA). Per ulteriori informazioni, vedere la sezione **Strumenti e informazioni sul rilevamento e sulla distribuzione** nel presente bollettino.

-   **Verifica della versione dei file**

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi su **Cerca**.
    2.  Nel riquadro **Risultati ricerca**, scegliere **Tutti i file e le cartelle** in **Ricerca guidata**
    3.  Nella casella **Nome del file o parte del nome**, digitare il nome del file, individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Cerca**.
    4.  Nell'elenco dei file, fare clic con il pulsante destro del mouse sul file individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Proprietà**.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.
    5.  Nella scheda **Versione** individuare il numero di versione del file installato nel sistema e confrontarlo con quello specificato nell'appropriata tabella di informazioni sui file.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.

-   **Controllo delle chiavi del Registro di sistema**

    Per verificare i file installati dall'aggiornamento per la protezione, è anche possibile esaminare le chiavi del Registro di sistema elencate nella **Tabella di riferimento** in questa sezione.

    Queste chiavi del Registro di sistema potrebbero non contenere un elenco completo dei file installati. Inoltre, se un amministratore o un OEM integra o esegue lo slipstreaming dell'aggiornamento per la protezione nei file di origine dell'installazione di Windows, è possibile che queste chiavi del Registro di sistema non vengano create correttamente.

**Verifica dell'applicazione dell'aggiornamento**

-   **Microsoft Baseline Security Analyzer**

    Per verificare se un aggiornamento per la protezione è stato applicato al sistema interessato, è possibile utilizzare lo strumento Microsoft Baseline Security Analyzer (MBSA). Per ulteriori informazioni, vedere la sezione **Strumenti e informazioni sul rilevamento e sulla distribuzione** nel presente bollettino.

-   **Verifica della versione dei file**

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi su **Cerca**.
    2.  Nel riquadro **Risultati ricerca**, scegliere **Tutti i file e le cartelle** in **Ricerca guidata**
    3.  Nella casella **Nome del file o parte del nome**, digitare il nome del file, individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Cerca**.
    4.  Nell'elenco dei file, fare clic con il pulsante destro del mouse sul file individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Proprietà**.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.
    5.  Nella scheda **Versione** individuare il numero di versione del file installato nel sistema e confrontarlo con quello specificato nell'appropriata tabella di informazioni sui file.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.

-   **Controllo delle chiavi del Registro di sistema**

    Per verificare i file installati dall'aggiornamento per la protezione, è anche possibile esaminare le chiavi del Registro di sistema elencate nella **Tabella di riferimento** in questa sezione.

    Queste chiavi del Registro di sistema potrebbero non contenere un elenco completo dei file installati. Inoltre, se un amministratore o un OEM integra o esegue lo slipstreaming dell'aggiornamento per la protezione nei file di origine dell'installazione di Windows, è possibile che queste chiavi del Registro di sistema non vengano create correttamente.

    Poiché esistono varie edizioni di Microsoft Windows, le operazioni descritte di seguito potrebbero risultare diverse nel sistema in uso. In tal caso, consultare la documentazione del prodotto per eseguire tutti i passaggi.

    1.  Fare clic su **Start**, quindi su **Cerca**.
    2.  Nel riquadro **Risultati ricerca**, scegliere **Tutti i file e le cartelle** in **Ricerca guidata**
    3.  Nella casella **Nome del file o parte del nome**, digitare il nome del file, individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Cerca**.
    4.  Nell'elenco dei file, fare clic con il pulsante destro del mouse sul file individuato nell'appropriata tabella di informazioni sui file, quindi fare clic su **Proprietà**.
        **Nota** A seconda dell'edizione del sistema operativo o dei programmi installati nel sistema, è possibile che alcuni dei file elencati nella tabella di informazioni sui file non siano installati.
    5.  Nella scheda **Versione** individuare il numero di versione del file installato nel sistema e confrontarlo con quello specificato nell'appropriata tabella di informazioni sui file.
        **Nota** È possibile che durante l'installazione alcuni attributi diversi dalla versione del file vengano modificati. Il confronto di altri attributi dei file con le informazioni disponibili nella tabella non è un metodo supportato per la verifica dell'applicazione dell'aggiornamento. In alcuni casi, inoltre, è possibile che i file vengano rinominati durante l'installazione. Se le informazioni sul file o sulla versione non sono disponibili, utilizzare uno degli altri metodi disponibili per verificare l'installazione dell'aggiornamento.

    Queste chiavi del Registro di sistema potrebbero non contenere un elenco completo dei file installati. Inoltre, se un amministratore o un OEM integra o esegue lo slipstreaming dell'aggiornamento per la protezione nei file di origine dell'installazione di Windows, è possibile che queste chiavi del Registro di sistema non vengano create correttamente.
